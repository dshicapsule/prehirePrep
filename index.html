Get oriented · Section 1

Overview

This automation moves new hires from our applicant system (Fountain) into our HR system (Rippling), and separately decides rehire eligibility for former employees.

What it does When a candidate is ready to start, the system gathers their details, creates their employee record in Rippling, records the result, moves them through the Fountain stages automatically, and writes their Rippling IDs back to Fountain — with no manual data entry. A second, independent job reviews former employees and stamps whether they can be rehired.

The 60-second summary

  1. Fountain holds candidates. When someone reaches “Ready to start,” they’re a candidate for automation.
  2. A Prep job pulls those candidates hourly into a Google Sheet with everything Rippling needs.
  3. A Draft Hire job creates each person’s Rippling record, logs every result in the permanent Draft Hire Log tab, and moves successes to Prehire Prep.
  4. On each person’s start date, they’re moved on to Prehire Tasks. Once they’re live in Rippling, their Rippling IDs are written back onto their Fountain profile.
  5. Separately, an hourly Rehire Eligibility job reviews former employees and writes their eligibility back into Fountain.

The golden rule Prep tabs get wiped and rebuilt every hour. The Draft Hire Log tab is never wiped — it is the permanent memory of who was hired and what happened. When in doubt, trust the Log.

What the full pipeline does — and what it writes, where

One hourly trigger runs runHourlyPipeline, which executes every step in order inside a single run (so steps never overlap). Two moves run on their own triggers. Click any step to see exactly what it appends and on which tab and column.

Reads the recomputed API tab and creates each person's draft hire record in Rippling. Blocked rows (any #N/A) are skipped with alert-D5.

For each successful draft hire still on [System] Ready to start, advances them one Fountain stage. Anyone who has drifted off that stage is skipped silently (no alert).

6Rehire review · runRehireEligibility▶

Independent job: pulls former employees on the Rehire Review stage and computes eligibility.

Rebuilds Rehire Review · cols A:N; eligibility is computed by formula in col R and written back to Fountain.

+Separate triggers — not in the hourly run▶

Daily ~7 AM · advanceToPrehireTasksOnStartDate — moves today's starters from Prehire Prep to Prehire Tasks. Stamps Draft Hire Log · col N = "Yes" and col R = "[System] Prehire Tasks".

Daily check

Most days the system runs itself and your job is a two-minute look.

Your morning check

  1. Check your email for alerts. The system emails the HR/Ops alerts inbox only when something needs attention — subject lines start with “ACTION REQUIRED.” No email = healthy run.
  2. Open the Draft Hire Log tab. New people should show success in the Status column (G). Anyone with error needs a look — see Troubleshooting.
  3. Confirm the moves happened. Every success should show Yes in column M (Moved to Prehire Prep). Anyone starting today should show Yes in column N (Moved to Prehire Tasks).
What “healthy” looks like No alert emails · new Log rows show success with column M = Yes · today's starters show Yes in column N · once someone is live in Rippling, column Q flips to Yes.

The two flows · Flow A

Draft Hire pipeline

The main flow: it turns a “Ready to start” candidate into a Rippling employee record, moves them to Prehire Prep when the hire succeeds, to Prehire Tasks on their start date, and writes their Rippling IDs back to Fountain once they go live.

Where to watch it Everything lands in the Google Sheet. Prep uses the Applicants – Draft Hire API Prep tab; the import reads Rippling | Draft Hire API; all results land in Draft Hire Log.

Stage 1 — Prep pull

Every hour, Prep finds everyone on a “Ready to start” stage in Fountain and writes one row per person — name, contact info, title, start date, pay, department, work location, SSN — into the Applicants – Draft Hire API Prep tab. When someone has more than one orientation slot (e.g. a reschedule), only their latest slot is used, so each person appears exactly once.

This tab is rebuilt every hour Don’t hand-edit it expecting changes to stick. Fix data in Fountain instead.

Stage 2 — The API tab

The Rippling | Draft Hire API tab uses formulas to reshape the prepped data into exactly the fields Rippling expects — title, compensation, employment type, work location, legal entity, weekly hours, and so on. It updates automatically; no one edits it by hand.

Stage 3 — Draft Hire import

The import sends each person to Rippling to create their draft hire (employee record), gets back their IDs, and records every outcome — success or error — in the Draft Hire Log.

Safe to re-run Anyone already in the Log as a success is skipped, so re-running never creates duplicates.
The TEST_MODE switch When TEST_MODE is true, the import only previews — it creates nothing and writes nothing to the Log. If nothing is appearing in the Log, check this first.

Stage 4 — The Draft Hire Log

This tab is the permanent record. Everyone sent to Rippling appears here exactly once — if the same person is processed again, their existing row is updated in place (matched by Fountain Applicant ID), so an error row flips to success once the problem is fixed.

Stage 5 — Move to Prehire Prep

The moment the import finishes, it moves every success one stage — from Ready to start to Prehire Prep — and stamps column M with Yes. This pulls a drafted person off the Prep tab on the next hourly rebuild, so they’re never sent to Rippling twice.

Why here, not later The Rippling permanent & profile numbers aren’t known until the person is hired in Rippling and picked up by the census refresh. Rather than wait for those, we move on draft-hire success alone, keeping the Prep tab clean.
Stage-gated — it checks Fountain first Before moving anyone, the sweep reads their live stage in Fountain and records it in column R. It only advances a person whose current stage is [System] Ready to start. If someone has since been put On Hold, rejected, or moved elsewhere, they’re skipped quietly — column R shows where they actually are and no alert is sent. This is why a stuck-looking row is often just someone Fountain has moved on its own.

Stage 6 — The 7 AM sweep (move to Prehire Tasks)

Each morning at 7:00 AM ET, anyone who is a success, already in Prehire Prep (column M = Yes), not yet moved (column N blank), and starting today is moved from Prehire Prep to Prehire Tasks, and column N is stamped Yes. No Fountain data is written here.

Re-run safe Column N prevents double-moving. Running the sweep twice in a day skips anyone already moved.
Stage-gated here too This move only fires when the person’s live Fountain stage (column R) is [System] Prehire Prep. Anyone not on that stage is skipped quietly with no alert — column R shows their real stage.

Stage 7 — Write Rippling IDs back to Fountain

Separately, every hour the three Rippling IDs (role / permanent / profile) are written onto the Fountain profile — but only when column O reads Yes (the formula turns it Yes once all three IDs are present, i.e. the person is live in Rippling and picked up by the census) and column Q is still blank. After writing, column Q is stamped Yes, so each hire is written back exactly once.

The two flows · Flow B

Rehire Eligibility

A separate hourly job. It reviews former employees on a special Fountain stage, works out whether they can be rehired, writes that answer back into Fountain, and moves them forward. It doesn’t touch the Draft Hire pipeline.

Where to watch it The Rehire Review tab in the same Google Sheet. Rows refresh every hour.

How it works

  1. Find the reviewers. The job looks up every “[System] Rehire Review API” stage across all funnels.
  2. Pull the people. It writes one row each (name, department, SSN, etc.) to the Rehire Review tab.
  3. Wait for the formulas. Column R computes the answer — one of Not Eligible, Rehire Eligible, or New Hire.
  4. Write it back. For each person with a valid answer, it saves the eligibility (and any Rippling IDs) onto their Fountain profile.
  5. Advance them. After a successful write-back, it moves the person to the next stage.
It only acts on clean answers If column R is blank, still calculating, or shows an error (like #N/A), that person is skipped and picked up on the next run. It never guesses.
Self-healing by design The job moves people out of the review stage as it processes them, so a large backlog simply clears a page at a time across hourly runs — which also keeps it under Fountain’s rate limits.

Setting or checking a trigger

  1. Open the Sheet → Extensions → Apps Script, then click the clock icon (Triggers) in the left rail.
  2. You need only two triggers: runHourlyPipeline on Time-driven → Hour timer → Every hour, and runDailyStartDateMoves on Time-driven → Day timer → 6am–7am. (If you previously set a daily trigger on advanceToPrehireTasksOnStartDate, repoint it to runDailyStartDateMoves so rehires are covered too.) To add one: + Add Trigger → choose the function → set the interval.
Why one hourly trigger, not several Apps Script’s “Every hour” option can’t be pinned to a specific minute, so separate hourly triggers could fire at the same time and collide on the Sheet. Running everything inside runHourlyPipeline forces a clean, sequential order instead.
Watch the 6-minute limit The hourly pipeline now chains census + prep + import + rehire intake/advance/write-back + rehire eligibility, plus a formula wait, all in one execution. Apps Script caps a single execution at ~6 minutes. If the log shows it timing out before the last step, tell me and we’ll trim the formula wait or split the chain.
Confirm the project timezone The 6 AM sweep depends on it. In the editor: Project Settings → Time zone should be America/New_York, or “6 AM” won’t mean 6 AM Eastern.
The step everyone forgets Apps Script runs the code pasted in the online editor, not the files on your computer. After changing any file you must paste it in and click Save — and if you edited two files, update both.
Run & fix · Section 5

Troubleshooting

Start with the symptom.

Common symptoms

Nothing is appearing in the Draft Hire Log

Most likely TEST_MODE is true in the import — it previews but writes nothing. Set it to false and re-run. If you get an alert-D8 email instead of rows, the updated post-process code may not be saved in the editor.

A person was hired but never moved forward in Fountain

First, check column R (Current Fountain Stage). The moves are stage-gated: a person only advances to Prehire Prep if Fountain shows them on [System] Ready to start, and only advances to Prehire Tasks if they’re on [System] Prehire Prep. If column R shows anything else (e.g. On Hold), the person was moved in Fountain and the script is correctly leaving them alone — this is expected, not a failure, and no alert is sent.

Stuck at Ready to start (column M blank) but R still says [System] Ready to start: the move should have happened. If M is blank on such a success row, the move failed — the next hourly import retries it. If it keeps failing, check for an alert email.

Stuck at Prehire Prep (column N blank): the 7 AM sweep needs all of — Status = success (G), column M = Yes, column N blank, start date = today (F), and column R = [System] Prehire Prep. If it’s not their start date, or they’re not on that stage, that’s expected.

Rippling IDs not on the Fountain profile (column Q blank): the write-back only fires once column O reads Yes, which needs all three IDs (J/K/L) present. Those don’t appear until the person is live in Rippling and picked up by the census, so a blank Q before then is normal.

An error row shows in the Log (Status = error)

Read the message in column I — it’s why Rippling or validation rejected the person (e.g. a missing required field). An error row was not created in Rippling, so it’s safe to retry: fix the data in Fountain / the Prep source, and the next run flips the same row to success (no duplicate).

Note: empty runs no longer clutter the Log. When nobody is “Ready to start,” the API tab computes to #N/A — those placeholder rows are now skipped, so you won’t see a bogus error row (e.g. “missing last_name”) and no alert email is sent.

A draft hire was blocked because of #N/A (alert-D5)

If a row has a real applicant but one or more of its mapped cells resolved to #N/A (or another spreadsheet error), the import will not create a partial profile. It blocks that person, writes an error row, and emails alert-D5 naming the person and the exact fields that were #N/A.

The usual cause is a lookup that didn’t map — most often department/division. Open the Rippling | Draft Hire API tab, find that row, and confirm the department/division (and any other looked-up value) exist in the Key tab so the mapping returns a value instead of #N/A. Once fixed, the next run imports the person automatically — no duplicate.

A row shows Status = review

review means the import could not match Rippling’s response back to that person, so it can’t tell whether the hire was created. To be safe it does not auto-retry (that’s how a created hire could be duplicated). Check Rippling directly: if the draft hire exists, change column G to success; if it doesn’t, change it to error. The row will then behave normally on the next run.

Rehire people aren’t being written back

The job only acts on an exact value in column R (Not Eligible, Rehire Eligible, or New Hire). If R is blank, still computing, or shows #N/A/#REF!, the person is skipped until the formula resolves. Confirm the column-R formula produces one of those exact values (spelling and case matter).

Reference & glossary

Glossary

Fountain
The applicant tracking system (ATS) where candidates live and move through stages.
Rippling
The HR system (HRIS) where the employee record is created.
Draft Hire
A new employee record created in Rippling via its Draft Hire API.
Stage
A step in a Fountain funnel (e.g. “Ready to start,” “Prehire Prep,” “Prehire Tasks”).
Applicant ID
Fountain’s unique ID for a person — the key used to match records everywhere.
Prep tab
A working tab that is wiped and rebuilt each hour.
Draft Hire Log
The permanent, never-wiped record of every hire result.
Write-back
Saving data (Rippling IDs, eligibility) from the Sheet back onto a Fountain profile.
Advance
Moving an applicant to the next Fountain stage.
TEST_MODE
A safety switch; when true, jobs read only and change nothing.
Trigger
A scheduled timer in Apps Script that runs a function automatically.
